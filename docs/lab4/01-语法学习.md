## 一、.tpp 文件：模板的声明与实现分离

### 1.1 为什么需要 .tpp 文件

在之前的实验中，模板类的声明和实现都写在同一个 `.h` 文件里。lab4 将两者拆开：

| 文件 | 内容 |
|------|------|
| `GachaPool.h` | 模板类的**声明**（类名、成员变量、成员函数签名） |
| `GachaPool.tpp` | 模板类的**实现**（每个成员函数的具体代码） |

```
GachaPool.h          ← 声明：class GachaPool { void addItem(...); ... }
GachaPool.tpp        ← 实现：template<T> void GachaPool<T>::addItem(...) { ... }
```

### 1.2 为什么不能像普通类一样用 .cpp

普通类（如 `Item`）可以拆成 `Item.h` + `Item.cpp`，分别编译后链接。但**模板不能**，因为：

> 模板的代码只有在**被实际使用时**（如 `GachaPool<Item>`）编译器才会生成真正的机器码。如果实现放在 `.cpp` 里，编译器在编译 `main.cpp` 时看不到实现，就无法生成代码，导致链接报错"undefined reference"。

### 1.3 解决方案：.tpp + 在 .h 末尾 include

`.tpp` 本质上还是头文件，只是用后缀名区分"这是模板实现"。关键在于 `.h` 末尾要 include 它：

```cpp
// GachaPool.h
#ifndef GACHA_POOL_H
#define GACHA_POOL_H

template<typename T>
class GachaPool {
public:
    void addItem(const T& item);   // 只有声明
    // ...
};

#include "GachaPool.tpp"   // ← 在文件末尾引入实现
#endif
```

```cpp
// GachaPool.tpp
#ifndef GACHA_POOL_TPP
#define GACHA_POOL_TPP

template<typename T>
void GachaPool<T>::addItem(const T& item) {   // 具体实现
    items_.push_back(item);
}
// ...
#endif
```

这样，任何 `#include "GachaPool.h"` 的文件都能同时拿到声明和实现，编译器就能正确生成模板代码。

### 1.4 .tpp 文件的函数定义语法

在 .tpp 中定义模板成员函数，需要在每个函数前加 `template<typename T>`，并用 `类名<T>::` 限定：

```cpp
// 普通类 (Item.cpp)
void Item::someMethod() { ... }

// 模板类 (GachaPool.tpp)
template<typename T>
void GachaPool<T>::addItem(const T& item) { ... }
```

## 二、异常处理：throw 与 try-catch

### 2.1 为什么需要抛异常

当函数遇到无法正常处理的情况时（如空池抽卡、数组越界），需要一种方式通知调用者"出错了"。有两种常见做法：

| 方式 | 示例 | 问题 |
|------|------|------|
| 返回特殊值 | `return -1` 表示出错 | 调用者可能忘记检查；-1 可能是合法值 |
| 抛异常 | `throw runtime_error(...)` | 强制调用者处理，不处理则程序终止 |

### 2.2 throw：抛出异常

```cpp
#include <stdexcept>   // runtime_error, out_of_range 等

// 抛出一个运行时异常
throw runtime_error("drawRandom from empty pool");

// 抛出一个越界异常
throw out_of_range("inventory index out of range");
```

常用的标准异常类（都在 `<stdexcept>` 中）：

| 异常类 | 含义 | 本实验中的使用场景 |
|--------|------|-------------------|
| `runtime_error` | 运行时错误 | GachaPool 空池时 drawRandom |
| `out_of_range` | 索引越界 | Inventory 访问无效索引 |

### 2.3 try-catch：捕获异常

调用可能抛异常的函数时，用 `try-catch` 包裹：

```cpp
try {
    Item item = pool.drawRandom();   // 可能抛 runtime_error
    cout << item << endl;
} catch (const runtime_error& e) {
    cout << "抽卡失败: " << e.what() << endl;
}
```

- `e.what()` 返回抛异常时传入的字符串（如 `"drawRandom from empty pool"`）
- 如果不 catch，异常会一直向上传递，最终导致程序终止并打印错误信息

### 2.4 本实验中不需要写 try-catch

本实验 focus 在**抛出**异常（`throw`），不需要写 `try-catch`。测试代码会通过间接方式验证异常是否被正确抛出。

## 三、enum class：强类型枚举

### 3.1 普通 enum vs enum class

lab3 使用了普通 enum：

```cpp
enum Rarity { R, SR, SSR };    // 值会"泄露"到外层作用域
```

lab4 新增了 `enum class`：

```cpp
enum class PoolType {
    NORMAL, FRIENDLY, PITY, ANTIPITY
};
```

区别：

| | `enum` | `enum class` |
|------|--------|-------------|
| 访问方式 | `R` 直接使用 | `PoolType::NORMAL` 必须带类名 |
| 隐式转 int | 可以（`if (rarity == 0)`）| 不可以，更安全 |
| 作用域 | 泄露到外层 | 限定在枚举名内 |
| 不同枚举比较 | 可以（容易出 bug）| 编译报错 |

### 3.2 使用方式

```cpp
// 声明
PoolType pool = PoolType::NORMAL;

// switch 判断（case 里不用写 PoolType::，编译器自动推导）
switch (pool) {
    case PoolType::PITY:    /* ... */ break;
    case PoolType::NORMAL:  /* ... */ break;
}

// 函数参数和返回值
PoolType selectPool() const;
Item drawFromPool(PoolType pool);
```

## 四、(void) 抑制未使用参数警告

在 student 的 TODO 骨架中会看到：

```cpp
void GachaSystem::loadDefaultItems() {
    // ---- BEGIN TODO 11 ----
    rPool_.addItem({"Test Item", R});  // 学生填写后 (void)item 会被替换
    // ---- END TODO 11 ----
}
```

而另一个例子：

```cpp
template<typename T>
void GachaPool<T>::addItem(const T& item) {
    // ---- BEGIN TODO 1 ----
    (void)item;   // ← 这一行
    // ---- END TODO 1 ----
}
```

`(void)item` 的作用是**告诉编译器"我故意不用这个参数，别报警告"**。因为 TODO 还没填写时，参数 `item` 没有被使用，编译器会报 `-Wunused-parameter`。一旦学生填入真正的实现（如 `items_.push_back(item)`），这行就会被替换。

> 你在 reference 代码中不会看到 `(void)`，因为那里所有参数都已被使用。
